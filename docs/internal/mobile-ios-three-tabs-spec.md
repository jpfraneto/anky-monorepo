# iOS Three-Tab App Architecture — Implementation Spec

This spec extends `mobile-ios-now-seed-spec.md`. That document covers identity, auth, writing UX, and the life/idle model. This document covers the three-tab structure, async prompt preloading, the "You" profile tab, the "Stories" tab, and the silent push notification integration.

## App Structure

The app has exactly three tabs at the bottom:

1. **Write** — the writing experience (already specified in `mobile-ios-now-seed-spec.md`)
2. **Stories** — cuentacuentos: stories generated from the parent's writing
3. **You** — what anky knows about the user, built from their writing history

The Write tab is the default. It is always the landing tab.

## Tab 1: Write

### Prompt Preloading

Before the user writes, the Write tab displays a personalized prompt. This prompt was generated by the server after the user's previous writing session.

#### On app launch

1. Check local cache for a stored prompt.
2. If cache exists and is less than 24 hours old, display it immediately.
3. In the background, call `GET /swift/v2/next-prompt` to check for a newer prompt.
4. If the server returns a newer prompt (compare `fromSession`), update the cache and the display.
5. If no cache exists, call the endpoint and display the result. If the call fails, display the default prompt.

#### Endpoint: `GET /swift/v2/next-prompt`

Headers:
```
Authorization: Bearer <session_token>
```

Response:
```json
{
  "prompt": "what lives underneath the thing you keep almost saying",
  "fromSession": "abc123-session-id"
}
```

- `prompt` — the writing prompt text. Always present. Lowercase, no period.
- `fromSession` — the writing session that generated this prompt. Nullable. Use this to detect staleness: if it matches the locally cached `fromSession`, no update needed.

If the user has never written, or the server has no personalized prompt yet, the response is:
```json
{
  "prompt": "close your eyes. take a deep breath. set a timer for 8 minutes. write without stopping. don't think. don't edit. just let it flow.",
  "fromSession": null
}
```

#### Prompt display

- Show the prompt text centered on the Write tab, above or below the `WRITE NOW` button.
- The prompt should feel like an invitation, not an instruction.
- Typography: lighter weight than the main UI. The prompt is ambient, not dominant.
- The `WRITE NOW` button and `8 minutes` label remain the primary visual elements.

#### Post-write prompt refresh

After every `POST /swift/v2/write` response, the server generates a new prompt in the background. The new prompt will be available within 10-30 seconds (Ollama generation time). The app does NOT need to poll — a silent push notification will arrive when it's ready.

### Writing Flow

No changes from `mobile-ios-now-seed-spec.md`. The write endpoint is `POST /swift/v2/write`. Short sessions (< 8 min or < 300 words) return `persisted: false`. Real ankys return `persisted: true` and spawn server-side pipelines.

After a successful write (any outcome), the response now includes a `spawned` object:
```json
{
  "ok": true,
  "sessionId": "uuid",
  "outcome": "anky",
  "wordCount": 412,
  "durationSeconds": 486.2,
  "flowScore": 0.83,
  "persisted": true,
  "spawned": {
    "ankyId": "uuid",
    "feedback": false,
    "cuentacuentos": true
  },
  "walletAddress": "0x1234...",
  "statusUrl": "/swift/v2/writing/uuid/status",
  "error": null
}
```

After the response arrives:
1. If `persisted` is true and `outcome` is `"anky"`, show the successful end state.
2. Optionally poll `statusUrl` to show pipeline progress (anky image generating, cuentacuentos generating).
3. Wait for the silent push to refresh the next prompt.

## Tab 2: Stories

This tab displays cuentacuentos — stories generated from the user's writing, set in the Ankyverse.

### Stories tab states

**Empty state** (no stories yet):
- "your writing becomes stories. write your first anky to begin."
- A subtle link or animation pointing to the Write tab.

**Stories available**:
- Show the most recent unplayed story at the top as the "ready" story.
- Below it, a scrollable list of past stories (played and unplayed).

### Ready story

Call `GET /swift/v2/cuentacuentos/ready` on tab appear.

Headers:
```
Authorization: Bearer <session_token>
```

Optional query parameter:
```
?childId=uuid
```
If set, scopes the story to a specific child profile. If omitted, returns the parent's stories.

Response (story available):
```json
{
  "id": "uuid",
  "writingId": "uuid",
  "parentWalletAddress": "0x...",
  "childWalletAddress": null,
  "title": "The Weaver of Primordia",
  "content": "Full story text...",
  "guidancePhases": [
    {
      "phase_type": "narration",
      "content": "In the kingdom of Primordia...",
      "narration": "In the kingdom of Primordia...",
      "duration_seconds": 45,
      "image_url": "https://anky.app/data/images/abc123.webp",
      "narration_es": "En el reino de Primordia...",
      "narration_zh": "...",
      "narration_hi": "...",
      "narration_ar": "..."
    }
  ],
  "played": false,
  "generatedAt": "2026-03-19T12:00:00Z",
  "chakra": 3,
  "kingdom": "Primordia",
  "city": "Emblazion",
  "contentEs": "Spanish translation...",
  "contentZh": "Chinese translation...",
  "contentHi": "Hindi translation...",
  "contentAr": "Arabic translation..."
}
```

Response (no stories): `null`

### Story playback

Each story has `guidancePhases` — an array of phases that form a guided storytelling experience:

- Each phase has `phase_type` (usually `"narration"`), `content`/`narration` (the text), `duration_seconds`, and optionally `image_url`.
- Display the story phase by phase. Show the image for each phase if available.
- Use `duration_seconds` to pace the transitions.
- Multilingual narrations are in `narration_es`, `narration_zh`, `narration_hi`, `narration_ar` fields within each phase. Use the device language to select the right one.
- The `content_es`, `content_zh`, `content_hi`, `content_ar` top-level fields contain the full story text in each language.

When the user finishes listening/reading the story:
```
POST /swift/v2/cuentacuentos/{id}/complete
Authorization: Bearer <session_token>
```
Response: `{ "ok": true }`

### Story history

Call `GET /swift/v2/cuentacuentos/history` for the full list.

Same auth and query params as `/ready`. Returns an array of story objects (same schema, but `guidancePhases` won't have `image_url` decorated — those are only on the `/ready` endpoint).

### Story assignment (optional, for child profiles)

If the user has child profiles, they can assign a story to a child:
```
POST /swift/v2/cuentacuentos/{id}/assign
Authorization: Bearer <session_token>
Content-Type: application/json

{
  "childWalletAddress": "0x..."
}
```

## Tab 3: You

This tab shows what anky has learned about the user from their writing practice.

### Endpoint: `GET /swift/v2/you`

Headers:
```
Authorization: Bearer <session_token>
```

Response:
```json
{
  "totalSessions": 42,
  "totalAnkys": 12,
  "totalWords": 28500,
  "currentStreak": 3,
  "longestStreak": 7,
  "bestFlowScore": 0.92,
  "avgFlowScore": 0.71,
  "psychologicalProfile": "A 400-word profile of who this person is based on their unfiltered writing...",
  "emotionalSignature": "2-3 sentences describing the dominant emotional texture of this person's inner world...",
  "coreTensions": "3-5 internal contradictions that keep showing up in the writing...",
  "growthEdges": "3-5 areas where this person is actively struggling or evolving...",
  "honchoContext": "Honcho's real-time accumulated understanding of this person...",
  "lastUpdated": "2026-03-19T04:00:00Z"
}
```

All text fields (`psychologicalProfile`, `emotionalSignature`, `coreTensions`, `growthEdges`, `honchoContext`) are nullable. They build up progressively:

- After the first writing session: only stats fields are populated.
- After ~5 real ankys: `psychologicalProfile`, `coreTensions`, `growthEdges`, and `emotionalSignature` appear.
- `honchoContext` builds continuously with every writing session sent to Honcho.
- Profile fields update every 5th real anky session.

### You tab layout

**Top section: Stats**
- Current streak (days in a row)
- Total ankys written
- Total words written
- Best flow score
- Average flow score

Display these as compact, beautiful numbers. The streak is the most important — make it prominent.

**Middle section: Profile cards**

Show each non-null profile field as an expandable card:

1. **Who you are** — `psychologicalProfile`
2. **Your emotional signature** — `emotionalSignature`
3. **Your tensions** — `coreTensions`
4. **Where you're growing** — `growthEdges`

Each card shows a preview (first ~2 lines) and expands on tap to show the full text.

If `honchoContext` is non-null and different from `psychologicalProfile`, show it as a fifth card: **What anky sees right now** — this is the real-time evolving understanding, vs the periodic profile snapshots.

**Empty state** (new user, no profile yet):
- Show the stats section with zeros.
- Instead of profile cards, show: "write your first anky. then write four more. anky needs to hear you before it can see you."
- This is not an error state. It's an invitation.

### Refresh behavior

- Call `GET /swift/v2/you` on tab appear.
- Cache the response locally for 5 minutes.
- After each write completes, invalidate the cache so the next tab visit fetches fresh data.

## Silent Push Notifications

### Purpose

After a writing session, the server generates a personalized prompt in the background (takes 10-30 seconds). When it's ready, the server can send a silent push notification to wake the app and pre-fetch the new prompt.

### Device token registration

After successful auth (challenge/verify), register for remote notifications:

```swift
UIApplication.shared.registerForRemoteNotifications()
```

In `application(_:didRegisterForRemoteNotificationsWithDeviceToken:)`, send the token to the backend:

```
POST /swift/v2/device-token
Authorization: Bearer <session_token>
Content-Type: application/json

{
  "token": "hex-encoded-apns-device-token",
  "platform": "ios"
}
```

Response: `{ "ok": true }`

Call this:
- After every successful auth
- Whenever the device token changes (iOS can rotate it)

### Handling silent pushes

Register for background fetch in your app's capabilities.

In `application(_:didReceiveRemoteNotification:fetchCompletionHandler:)`:

1. Check the notification payload for a `type` field.
2. If `type` is `"prompt_ready"`, call `GET /swift/v2/next-prompt` and cache the result locally.
3. If `type` is `"profile_updated"`, invalidate the `/you` cache so it refreshes on next tab visit.
4. Call the completion handler with `.newData` or `.noData`.

The push payload will be:
```json
{
  "aps": {
    "content-available": 1
  },
  "type": "prompt_ready"
}
```

No alert, no badge, no sound. The user sees nothing. The app just pre-fetches.

### Background App Refresh (fallback)

Also register for Background App Refresh as a fallback for when silent pushes don't arrive:

```swift
UIApplication.shared.setMinimumBackgroundFetchInterval(
    UIApplication.backgroundFetchIntervalMinimum
)
```

In the background fetch handler:
1. Call `GET /swift/v2/next-prompt`
2. Cache the result
3. Call completion handler

This ensures the prompt is always fresh even if push delivery is unreliable.

## Children (existing, no changes)

Child profile endpoints remain as specified:

- `POST /swift/v2/children` — create child profile
- `GET /swift/v2/children` — list children
- `GET /swift/v2/children/{childId}` — get child detail

These are used to scope cuentacuentos stories to a specific child. The child management UI can live in the You tab or in a settings sub-screen.

## Complete Endpoint Reference

All endpoints use `Authorization: Bearer <session_token>` header unless noted.

| Method | Path | Purpose |
|--------|------|---------|
| POST | `/swift/v2/auth/challenge` | Request sign-in challenge (no auth) |
| POST | `/swift/v2/auth/verify` | Verify signature, get session token (no auth) |
| DELETE | `/swift/v2/auth/session` | Logout |
| GET | `/swift/v2/me` | User profile + stats |
| GET | `/swift/v2/writings` | Writing history |
| POST | `/swift/v2/write` | Submit writing session |
| GET | `/swift/v2/writing/{sessionId}/status` | Poll pipeline status |
| GET | `/swift/v2/next-prompt` | Get precomputed writing prompt |
| GET | `/swift/v2/you` | Full profile (stats + psychology + Honcho) |
| POST | `/swift/v2/device-token` | Register APNs device token |
| GET | `/swift/v2/children` | List child profiles |
| POST | `/swift/v2/children` | Create child profile |
| GET | `/swift/v2/children/{childId}` | Get child profile |
| GET | `/swift/v2/cuentacuentos/ready` | Next unplayed story |
| GET | `/swift/v2/cuentacuentos/history` | Story history |
| POST | `/swift/v2/cuentacuentos/{id}/complete` | Mark story as played |
| POST | `/swift/v2/cuentacuentos/{id}/assign` | Assign story to child |
| POST | `/swift/v1/admin/premium` | Toggle premium (admin) |

## Data Flow Summary

```
User opens app
  → load cached prompt (instant)
  → GET /swift/v2/next-prompt (background refresh)
  → display prompt on Write tab

User writes for 8+ minutes
  → POST /swift/v2/write
  → server responds immediately with persisted: true
  → server spawns async:
      1. anky image generation (Flux/ComfyUI)
      2. cuentacuentos story (Claude + Ollama)
      3. next prompt generation (Ollama + Honcho context)
      4. memory pipeline (profile update every 5th anky)
      5. Honcho send (continuous peer modeling)

  → ~15 seconds later: server sends silent push
  → app wakes in background
  → app calls GET /swift/v2/next-prompt
  → app caches new prompt
  → next time user opens app: new prompt is already there

User opens Stories tab
  → GET /swift/v2/cuentacuentos/ready
  → story plays with images and narration
  → POST /swift/v2/cuentacuentos/{id}/complete

User opens You tab
  → GET /swift/v2/you
  → stats + profile cards displayed
  → profile gets richer with every write
```

## Implementation Order

1. Three-tab navigation structure (Write / Stories / You)
2. Write tab: integrate `GET /swift/v2/next-prompt` with local caching
3. You tab: call `GET /swift/v2/you`, display stats + profile cards with empty states
4. Stories tab: call `/cuentacuentos/ready`, implement phase-by-phase playback
5. Device token registration: `POST /swift/v2/device-token` after auth
6. Silent push handling: background fetch of `/next-prompt` on notification
7. Background App Refresh fallback
8. Story history and assignment
9. Cache invalidation after writes

## What Not To Build

- No social features. No feed. No followers.
- No in-app purchase flow (premium is toggled server-side via admin).
- No push notification permission prompt for alerts — we only use silent pushes. The user should never see a notification from anky.
- No manual refresh buttons. Everything updates automatically via silent push + background fetch.
- No loading spinners on the You tab. Show cached data immediately, update in background.
